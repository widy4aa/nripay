# 🏃‍♂️ Sprint 3: Chain Visualizer, Rollback Engine & Top-up / Online Transfer
**Durasi:** 2 Minggu (14 Hari Kerja) | **Target Utama:** Rekonsiliasi percabangan rantaian (*Chain Fork & Cascade Rollback*), Visualisasi Rantai di Dashboard (`Chain Visualizer`), serta alur Top-up VA & Transfer Online.

---

## 🎯 Tujuan Sprint

Sprint 3 mengamankan ekosistem dari anomali pembelahan rantai offline (*chain fork* / *double-spend*) melalui mekanisme eksekusi *rollback* atomik dan memberikan visibilitas penuh kepada Admin dalam melacak asal-usul setiap token CBDC dari `Hop 0` sampai `Hop 3`.

1. **Client App (Flutter):** Implementasi pengeksekusi antrian penyesuaian saldo lokal (`RollbackQueue` executor), layar penjelas penyesuaian saldo (`balance_adjusted_page.dart`), alur Top-up Virtual Account/QRIS, serta pengiriman uang online via Wallet ID.
2. **Server (Backend NestJS):** Pembangunan modul `ChainService` (rekonstruksi *linked-list chain*) dan penyempurnaan `ReconcileService` untuk menangani *Cascade Rollback*. Implementasi modul `Top-up` + *Webhook Handler*, serta `Transfer Online`.
3. **Dashboard (Next.js):** Pembuatan fitur visualisasi rantai interaktif (`Chain Visualizer`), pemantau distribusi lompatan offline (`Hop Chain Tracker`), serta antrean pemeriksaan KYC pengguna (`KYC Queue`).

## 📱 1. Client App (Flutter / Android)

### A. Lingkup Tugas & Deliverables
- [ ] **Modul Eksekusi Rollback Lokal (`rollback_queue` & `balance_adjusted_page.dart`):**
  - Pembuatan *executor service* yang berjalan setiap kali aplikasi dibuka atau sinkronisasi selesai: mengambil baris `RollbackQueue` di mana `is_executed = false`.
  - Penanganan 3 jenis penyesuaian lokal dalam transaksi SQLite atomik:
    1. `RELEASE_RESERVE` (SEND ditolak): lepaskan kunci saldo (`reserved_cent -= amount`). Uang tidak pernah keluar.
    2. `DEDUCT_RECEIVED` (RECEIVE ditolak): tarik kembali saldo sementara (`amount_cent -= amount`).
    3. `CASCADE_DEDUCT`: tarik saldo akibat transaksi induk dalam rantai ditolak oleh server.
  - Setelah eksekusi sukses (`is_executed = true`), tampilkan modal layar penuh `balance_adjusted_page.dart` yang menjelaskan dengan bahasa manusia ramah pengguna mengapa saldonya disesuaikan, lengkap dengan tombol `Ajukan Banding / Klaim`.
- [ ] **Modul Top-up Saldo (`topup_page.dart` & `topup_va_page.dart`):**
  - Pilihan metode pengisian: Virtual Account (BCA, BRI, Mandiri) dan QRIS.
  - Memanggil `POST /api/v1/topup/create` dan melakukan *polling* ringan / mendengarkan *push notification* hingga status berubah menjadi `CONFIRMED`.
- [ ] **Modul Transfer Online (`online_transfer_page.dart`):**
  - Pengiriman dana langsung via server untuk pengguna yang sedang terhubung ke internet.
  - Resolusi nama penerima real-time menggunakan `GET /api/v1/wallet/resolve/:identifier`.
  - Memanggil `POST /api/v1/transfer`, menyertakan PIN yang divalidasi lokal/server, dan memperbarui saldo seketika.

## ⚙️ 2. Server (Backend NestJS + Prisma)

### A. Lingkup Tugas & Deliverables
- [ ] **Penyempurnaan Rekonsiliasi CBDC Core (`src/cbdc-core/*`):**
  - **`chain.service.ts`:** Mengambil seluruh transaksi dari `global_ledger` berdasarkan `mint_tx_id` yang sama. Menelusuri *linked list* dari `parent_tx_id` ke `tx_id`. Mendeteksi *Chain Fork* jika terdapat 2 transaksi di `hop_count` yang sama pada pohon yang sama.
  - **`reconcile.service.ts` (Cascade Rollback Engine):**
    - Jika transaksi X ditolak (*double spend* / *fork*), cari secara rekursif seluruh anak rantainya (`SELECT * FROM global_ledger WHERE parent_tx_id = X.tx_id`).
    - Tandai semua anak rantai sebagai `REJECTED` dengan alasan `CASCADE_PARENT_REJECTED`.
    - Susun instruksi penyesuaian di *response payload* (`requires_client_rollback = true`, dan daftar `rollback_items`) agar klien yang terkena dampak melakukan pemotongan saldo saat sinkronisasi berikutnya.
- [ ] **Modul Top-up & Webhook (`/api/v1/topup/*` & `/api/v1/mock-bank/webhook`):**
  - `POST /topup/create`: Buat entri `topup_requests` dengan nomor Virtual Account unik (`{bank_code}{YYYYMMDD}{random_6_digit}`).
  - `POST /mock-bank/webhook`: Simulator konfirmasi pembayaran VA. Memverifikasi HMAC rahasia, mengubah status menjadi `CONFIRMED`, memanggil `MintService.mint()` untuk menerbitkan token baru ke `global_ledger` (`hop = 0`), dan memperbarui `amount_cent` pengguna.
- [ ] **Modul Transfer Online (`POST /api/v1/transfer`):**
  - Memeriksa PIN pengguna, memvalidasi kecukupan saldo (`amount_cent - reserved_cent >= amount`), memotong saldo pengirim & menambah saldo penerima secara atomik dalam satu transaksi Prisma, serta mencatat ke `global_ledger` (`transfer_medium = 'ONLINE'`).

## 🧪 4. Kriteria Pengujian & Verifikasi Sprint 3

### A. Pengujian Skenario Double-Spend & Cascade Rollback (Skenario Terberat)
1. **Setup Awal:** Top-up **User A** sebesar **Rp 200.000**.
2. **Kirim Offline Legitimasi:** User A kirim NFC **Rp 100.000** ke **User B** (`tx-AAA`, Hop 1). User B kirim NFC **Rp 80.000** ke **User C** (`tx-BBB`, Hop 2). User C kirim NFC **Rp 60.000** ke **User D** (`tx-CCC`, Hop 3).
3. **Double-Spend Kecurangan:** Di HP **User A**, kirim lagi token awal yang sama (Top-up Rp 200.000) sebesar **Rp 100.000** ke **User E** (`tx-DDD`, Hop 1).
4. **First-Sync-Wins:** Hubungkan **User E** ke internet duluan dan lakukan sinkronisasi. Transaksi `tx-DDD` diterima server (`status = SYNCED`). Saldo User E bertambah Rp 100.000.
5. **Eksekusi Rollback Cascade:**
   - Hubungkan **User B** ke internet dan klik `Sync`. Server mendeteksi bahwa `mint_tx_id` pada Hop 1 telah diklaim oleh `tx-DDD`. Server menolak `tx-AAA` (`CHAIN_FORK`) dan menolak pengirimannya `tx-BBB` (`CASCADE_PARENT_REJECTED`).
   - HP User B menerima respons `requires_client_rollback = true`. Aplikasi mengeksekusi `RollbackQueue`, memotong saldo menjadi Rp 0, dan memunculkan modal peringatan `balance_adjusted_page.dart`.
   - Hubungkan **User C** dan **User D** untuk sinkronisasi. Keduanya menerima status `CASCADE_PARENT_REJECTED` dan saldonya disesuaikan secara otomatis.

### B. Verifikasi Visualisasi Dashboard
1. Buka halaman Dashboard di `/ledger/chain/<uuid-mint-topup-user-a>`.
2. Pastikan diagram grafis menunjukkan dengan tepat:
   - Cabang Hop 1-A (`tx-DDD` ke User E) berstatus hijau (`SYNCED`).
   - Cabang Hop 1-B (`tx-AAA` ke User B) berstatus merah (`REJECTED - CHAIN_FORK`).
   - Anak cabang Hop 2 (`tx-BBB` ke User C) dan Hop 3 (`tx-CCC` ke User D) berstatus merah (`REJECTED - CASCADE_PARENT_REJECTED`).
