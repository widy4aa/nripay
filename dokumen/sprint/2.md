# 🏃‍♂️ Sprint 2: Offline CBDC Engine, NFC Transfer & Sync V1
**Durasi:** 2 Minggu (14 Hari Kerja) | **Target Utama:** Komunikasi APDU NFC HCE antar perangkat, kriptografi Ed25519 lokal (5 Lapis Verifikasi Penerima), Rekonsiliasi Server Batch Sync V1, dan Simulator Bank Sentral (`mock-bank`).

---

## 🎯 Tujuan Sprint

Sprint 2 merealisasikan fitur unik dan paling kritis dari Nirpay: kemampuan pengiriman token CBDC secara **offline tanpa koneksi internet** melalui NFC, serta rekonsiliasinya ke server saat perangkat kembali online.

1. **Client App (Flutter):** Implementasi nyata kriptografi Ed25519 menggunakan *package* `cryptography`, penyempurnaan *MethodChannel* Android HCE untuk bertukar payload APDU secara stabil, penegakan **5 Lapis Verifikasi LOKAL** di perangkat penerima sebelum menambah saldo, serta pembentukan paket `POST /api/v1/sync`.
2. **Server (Backend NestJS):** Pembangunan modul `cbdc-core` (Signature Service & Ledger Service), pengaktifan modul `mock-bank` untuk *minting* awal CBDC dengan tanda tangan Ed25519, serta implementasi endpoint `POST /api/v1/sync` V1 (verifikasi kriptografi + pengecekan *double-spend*).
3. **Dashboard (Next.js):** Pembuatan halaman `Global Ledger` (`/ledger`), form pengisian CBDC manual (`Manual Mint`), dan halaman detail pengguna dengan tab riwayat transaksi.

## 📱 1. Client App (Flutter / Android)

### A. Lingkup Tugas & Deliverables
- [ ] **Modul Kriptografi Lokal (`CryptoService`):**
  - Implementasi `verifyBankSignature()` (Ed25519) untuk memastikan token sah dari Bank Pusat.
  - Implementasi `verifySenderSignature()` (Ed25519) untuk memverifikasi kepemilikan token dari pengirim.
  - Implementasi `signData()` menggunakan *private key* dari TEE saat bertindak sebagai pengirim (*Sender*).
- [ ] **Penyempurnaan NFC HCE Native Android (`android/app/src/main/kotlin/...`):**
  - Pemantapan protokol komunikasi APDU (ISO-DEP 14443-4) antara *HostApduService* (Pengirim/HCE) dan *NfcManager* (Penerima/Reader).
  - Penanganan paket data besar (*chunking/fragmentation*) jika payload JSON tersertifikasi melebihi batas APDU tunggal (255 byte).
- [ ] **Modul Transaksi Offline (`nfc_transfer_page.dart` & `receive_money_page.dart`):**
  - **Pengirim (HCE):** Validasi lokal pra-kirim (`hop_count < max_hop` dan `spendable_cent >= amount`). Jika lolos, kunci saldo dalam satu transaksi Drift atomik (`reserved_cent += amount`, `hop_count += 1`), lalu kirim APDU.
  - **Penerima (Reader):** Penegakan **5 Lapis Verifikasi Lokal**:
    1. *Bank Signature* valid?
    2. *Sender Signature* valid?
    3. Waktu tidak kedaluwarsa (`now() < expires_at`)?
    4. Batas *hop* belum terlampaui (`payload.hop_count < max_hop`)?
    5. Anti *Double-Spend/Replay* lokal: `tx_id` belum ada ATAU `(mint_tx_id, hop_count)` belum pernah dicatat di tabel `Transactions` Drift?
  - Jika 5 verifikasi lolos: Eksekusi transaksi Drift atomik (`amount_cent += amount`, `pending_receive_hop = MAX(pending_receive_hop, hop_count)`, insert ke `Transactions` dengan `sync_status = 'PENDING'`).
- [ ] **Modul Sinkronisasi V1 (`sync_service.dart`):**
  - Mengumpulkan semua transaksi Drift di mana `sync_status = 'PENDING'` dan mengirimkannya ke `POST /api/v1/sync`.
  - Memproses respons server per-transaksi: jika `SYNCED`, sesuaikan `amount_cent` ke nilai `server_balance_after` dan reset `hop_count = 0`.

## ⚙️ 2. Server (Backend NestJS + Prisma)

### A. Lingkup Tugas & Deliverables
- [ ] **Modul CBDC Core (`src/cbdc-core/*`):**
  - **`signature.service.ts`:** Menggunakan *package* `noble-ed25519` untuk memverifikasi `bank_signature_b64` dan `sender_signature_b64`.
  - **`ledger.service.ts`:** Mengelola operasi CRUD pada tabel `global_ledger`. Menegakkan *UNIQUE constraint* `(mint_tx_id, hop_count)` sebagai proteksi tingkat server.
- [ ] **Simulator Bank Sentral (`src/mock-bank/*`):**
  - **`bank-signature.service.ts`:** Membuka Ed25519 *private key* dari konfigurasi (`BANK_PRIVATE_KEY_B64`) dan menandatangani payload awal (`amount_cent`, `currency`, `receiver_id`, `mint_tx_id`, `hop_count = 0`).
  - `POST /api/v1/mock-bank/mint`: Endpoint untuk menerbitkan token CBDC langsung ke dompet pengguna (digunakan untuk pengujian & top-up).
- [ ] **Endpoint Rekonsiliasi V1 (`POST /api/v1/sync`):**
  - Menerima *batch pending transactions* dari pengguna yang baru terhubung ke internet.
  - Untuk setiap transaksi dalam *array*:
    1. Verifikasi Ed25519 *signature* bank & pengirim.
    2. Periksa masa berlaku (`expires_at`).
    3. Periksa apakah `tx_id` sudah ada di `global_ledger` (*double spend*).
    4. Periksa apakah `(mint_tx_id, hop_count)` sudah terdaftar (*chain fork* / *replay attack*).
    5. Periksa kecukupan saldo pihak pengirim di server.
  - Jika lolos: masukkan ke `global_ledger` dengan status `SYNCED`, perbarui saldo pengirim dan penerima di `wallet_balances`, dan kembalikan `server_balance_after`.
  - Jika gagal: catat alasan penolakan (`reject_reason` = `DOUBLE_SPEND`, `CHAIN_FORK`, `INSUFFICIENT_BALANCE`, `SIG_INVALID`, `EXPIRED`), dan kembalikan status `REJECTED`.

## 🧪 4. Kriteria Pengujian & Verifikasi Sprint 2

### A. Pengujian End-to-End Offline (2 Perangkat Android / Emulator HCE)
1. Buka Dashboard di `http://localhost:3001/mint`. Terbitkan CBDC sebesar **Rp 500.000** (`50.000.000 cents`) ke **User A**.
2. Buka aplikasi Flutter di HP **User A**, jalankan sinkronisasi via tombol `Sync`. Saldo User A menjadi **Rp 500.000** (`hop_count = 0`).
3. **Matikan koneksi Wi-Fi dan Data Seluler pada HP User A dan User B.**
4. Di HP User A, pilih `Kirim` -> `NFC` -> Masukkan nominal **Rp 200.000**.
5. Di HP User B, pilih `Terima` -> `NFC`. Dekatkan kedua HP.
6. **Verifikasi Hasil Lokal:**
   - HP User A menyusut saldonya menjadi **Rp 300.000** (Rp 200.000 masuk ke `reserved_cent`).
   - HP User B bertambah saldonya menjadi **Rp 200.000** (`pending_receive_hop = 1`).
   - Cek database SQLite Drift di kedua HP: transaksi tercatat dengan status `PENDING`.

### B. Pengujian Sinkronisasi & First-Sync-Wins (Server Verification)
1. Hubungkan kembali HP **User B** ke internet. Tap `Sync`.
2. Verifikasi di Terminal/Console Backend bahwa server berhasil memverifikasi tanda tangan Ed25519 dan mencatat transaksi ke `global_ledger` (`status = SYNCED`, `hop = 1`).
3. Buka Dashboard Admin di `/ledger`. Pastikan transaksi transfer **Rp 200.000** dari User A ke User B muncul dengan jelas di daftar buku besar.
