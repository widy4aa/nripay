## рџЋЇ Tujuan Sprint

Sprint 4 memberikan kekuatan intervensi penuh kepada Admin untuk membekukan aktivitas mencurigakan atau mengoreksi saldo secara manual, namun diimbangi dengan akuntabilitas tinggi di mana pengguna Client dapat mengajukan banding atau klaim dengan menyertakan bukti konkret.

1. **Client App (Flutter):** Implementasi UI/UX pengajuan Klaim untuk transaksi yang ditolak (`claim_page.dart`), pengajuan Banding (`dispute_page.dart`) atas tindakan intervensi Admin lengkap dengan pengunggah bukti, serta penanganan status transaksi yang dibekukan (`FROZEN`).
2. **Server (Backend NestJS):** Pembangunan endpoint intervensi Admin (`POST /admin/transactions/:tx_id/freeze`, `POST /admin/users/:id/freeze`, `POST /admin/balance/:user_id/adjust`), sistem pencatatan `admin_actions`, serta modul CRUD Klaim (`claim_requests`) dan Banding (`disputes`).
3. **Dashboard (Next.js):** Pembuatan antarmuka kontrol Admin untuk membekukan transaksi (`Freeze Transaction`), mengelola daftar pembekuan aktif (`Active Freezes`), melakukan penyesuaian saldo (`Balance Adjustment`), serta meninjau dan memutuskan antrean Banding (`Dispute Queue`).

## вљ™пёЏ 2. Server (Backend NestJS + Prisma)

### A. Lingkup Tugas & Deliverables
- [ ] **Modul Kontrol Transaksi Admin (`/api/v1/admin/transactions/*` & `/api/v1/admin/users/*`):**
  - `POST /admin/transactions/:tx_id/freeze`: Kunci transaksi agar tidak diproses saat sinkronisasi klien (`global_ledger.status = 'FROZEN'`, insert `frozen_transactions`). Mendukung `freeze_type = 'HOLD'` (jeda sementara) dan `'FORCE_REJECT'` (tolak permanen beserta anak rantainya).
  - `POST /admin/transactions/:tx_id/unfreeze`: Kembalikan status dari `FROZEN` menjadi `SYNCED` atau aktif kembali.
  - `POST /admin/transactions/:tx_id/force-close`: Tutup paksa rantai dari node tersebut; semua anak rantai di bawahnya diubah menjadi `REJECTED` (`FORCE_CLOSED`).
  - `POST /admin/users/:id/freeze` & `/unfreeze`: Ubah status `is_active` pengguna dan catat ke `frozen_accounts`.
- [ ] **Modul Penyesuaian Saldo Admin (`POST /api/v1/admin/balance/:user_id/adjust`):**
  - Menambah (`CREDIT`) atau mengurangi (`DEBIT`) saldo `amount_cent` pengguna pada tabel `wallet_balances` secara atomik.
  - Wajib menyertakan alasan (`reason`) dan mencatat riwayat ke tabel `balance_adjustments` serta `admin_actions`.
- [ ] **Modul Klaim (`/api/v1/claim/*`) & Banding (`/api/v1/dispute/*`):**
  - `POST /claim` & `GET /claim/:id`: Pembuatan dan pelacakan klaim transaksi oleh pengguna.
  - `POST /dispute`: Pembuatan banding pengguna (`dispute_type` = `TRANSACTION_FROZEN`, `BALANCE_ADJUSTMENT`, `CLAIM_REJECTED`, `ACCOUNT_FROZEN`) lengkap dengan URL bukti lampiran.
  - `POST /admin/disputes/:id/review`: Endpoint khusus Admin untuk memutuskan banding (`ACCEPTED` / `PARTIAL_ACCEPTED` / `REJECTED`). Jika `ACCEPTED` atas potongan saldo, otomatis lakukan *refund* ke `wallet_balances` pengguna.
- [ ] **Audit Trail Admin (`admin_actions`):**
  - Setiap kali Admin memanggil endpoint kontrol atau penyesuaian, catat tindakan tersebut ke tabel `admin_actions` (append-only) sebagai bukti akuntabilitas.

## рџ§Є 4. Kriteria Pengujian & Verifikasi Sprint 4

### A. Pengujian Interupsi Pembekuan Transaksi (Freeze Tx Flow)
1. User A mengirimkan transaksi offline NFC ke User B (`tx-ghi-789`).
2. Sebelum User B sempat melakukan sinkronisasi, Admin membuka Dashboard di `/ledger/freeze` dan membekukan `tx-ghi-789` (`freeze_type = HOLD`, alasan: *"Investigasi kecurangan"*).
3. User B menghubungkan HP ke internet dan menekan tombol `Sync`.
4. **Verifikasi:** Server merespons dengan status transaksi `FROZEN`. Aplikasi Flutter User B tidak menambah atau mengurangi saldo, melainkan menampilkan ikon gembok kuning pada transaksi tersebut dan spanduk pemberitahuan di beranda.

### B. Pengujian Lapisan Kepercayaan (Balance Adjust -> Dispute -> Accept & Refund)
1. **Aksi Admin:** Admin membuka Dashboard di `/balance/adjust`, mencari **User C**, dan memotong saldonya (`DEBIT` sebesar **Rp 500.000** / `50.000.000 cents`) dengan alasan *"Koreksi sistem"*. Saldo User C berkurang dari Rp 1.000.000 menjadi Rp 500.000.
2. **Pengajuan Banding (Client):** User C melihat notifikasi pemotongan di aplikasi Flutter, lalu menekan tombol `Ajukan Banding`. User C mengisi judul *"Pemotongan tidak sah"*, menjelaskan kronologi, dan mengunggah foto bukti resi. Status banding menjadi `SUBMITTED`.
3. **Review & Resolusi (Dashboard):** Admin membuka `/disputes`, memilih banding User C, membaca penjelasan dan memeriksa foto bukti yang dilampirkan. Admin memilih keputusan `ACCEPT` dan memasukkan nominal refund **Rp 500.000**.
4. **Verifikasi Akhir:**
   - Saldo `amount_cent` User C di tabel `wallet_balances` otomatis kembali menjadi **Rp 1.000.000**.
   - Status banding menjadi `ACCEPTED`.
   - Tindakan awal dan pengembalian tercatat dengan rapi di `admin_actions` dan `balance_adjustments`.
